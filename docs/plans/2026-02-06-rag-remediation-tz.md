# ТЗ на исправление RAG-проекта (Docker + Ingestion + Qdrant + Telegram)

## 1. Цель
Довести проект до стабильного рабочего состояния по контуру:
Google Drive/`drive-sync` -> Ingestion -> Qdrant -> Telegram RAG Bot.

## 2. Область работ
- Docker-стек (`docker-compose.vps.yml`, сервисы `vps-*`)
- Ingestion pipeline (`src/ingestion/unified/*`)
- Telegram bot (`telegram_bot/*`)
- Тесты и покрытие
- Операционная документация (`CLAUDE.md`, план/реестр дефектов)

## 3. Что исправить (обязательно)

### CRITICAL
1. Стабилизировать полный E2E-контур ingestion -> retrieval -> bot answer.
2. Устранить все блокирующие падения unit/smoke, мешающие выпуску.
3. Привести тестовый запуск к воспроизводимому окружению на хосте (рабочий venv без битых интерпретаторных ссылок).

### HIGH
1. Закрыть несоответствие identity-флоу документов:
   - добавить и использовать `.gdrive_manifest.json`;
   - исключить дубли при rename/move.
2. Зафиксировать единый source-of-truth для деплоя:
   - синхронизировать `/home/admin/rag-fresh` и `/opt/rag-fresh`;
   - убрать риск «правки не в том репозитории».
3. Унифицировать коллекции Qdrant по всем сервисам/env:
   - бот, ingestion, smoke/e2e должны смотреть в одну целевую коллекцию.
4. Добиться устойчивого старта бота после сетевых сбоев (retry/backoff) и проверить на рестартах.

### MEDIUM
1. Добить покрытие тестами до целевого порога 80% и зафиксировать отчёт.
2. Обновить тесты под текущие контракты payload/state.
3. Привести документацию к фактическому runtime (vps-stack, env, healthchecks, команды проверки).

## 4. Уже внедрённые фиксы (не откатывать)
1. Ленивая загрузка в `src/__init__.py` для исключения зависаний CLI/импортов.
2. Ленивая загрузка в `src/ingestion/__init__.py` и `src/ingestion/unified/__init__.py`.
3. Retry/backoff при старте Telegram бота в `telegram_bot/main.py`.
4. Расширение state-upsert в ingestion:
   - добавлен `upsert_state_sync`;
   - сохранение `source_path/file_name/mime_type/file_size` до обработки.
5. Добавлены регрессионные тесты:
   - `tests/unit/test_lazy_imports.py`;
   - расширены `tests/unit/test_main.py`;
   - обновлён `tests/unit/ingestion/test_state_manager_sync.py`.

## 5. Критерии приёмки
1. E2E:
   - добавление файла в `drive-sync` -> появление чанков в Qdrant;
   - удаление файла -> удаление чанков из Qdrant;
   - Telegram-бот отвечает релевантно по этим документам.
2. Docker:
   - после перезапуска сервера сервисы стартуют без ручных правок;
   - healthchecks зелёные по всем core сервисам.
3. Тесты:
   - unit/smoke обязательные наборы проходят;
   - coverage >= 80% (сохранён артефакт/отчёт).
4. Документация:
   - обновлён `CLAUDE.md` (план, статус, проблемы, что закрыто);
   - актуален реестр дефектов с `open/in_progress/fixed/verified`.

## 6. Порядок выполнения
1. Закрыть CRITICAL (стабильный E2E + блокирующие тесты).
2. Закрыть HIGH (manifest, синхронизация репозиториев, коллекции).
3. Добить MEDIUM (coverage/docs).
4. Финальный повторный прогон полного контроля.

## 7. Формат отчёта по выполнению
Для каждого пункта:
- Статус: `open | in_progress | fixed | verified`
- Изменённые файлы
- Проверка (команда + результат)
- Остаточный риск (если есть)
