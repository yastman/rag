# ТЗ: Роль Orchestrator для RAG-проекта

## 1. Цель
Оркестратор ведет проект до полностью рабочего состояния по цепочке:
Google Drive -> Ingestion -> Qdrant -> Telegram RAG Bot.

## 2. Роль оркестратора
- Делает технический аудит (код + Docker + конфиги + runtime).
- Самостоятельно запускает проверки и фиксит найденные дефекты.
- Ведет план работ, отмечает статус задач, добавляет новые ошибки в реестр.
- После каждого цикла делает повторный ревью и повторную валидацию.
- Работает до состояния «всё стабильно работает», без остановки на полпути.

## 3. Обязательный цикл работ
1. Ревью текущего состояния (код, compose, env, зависимости, логи).
2. Фиксация багов в реестре (severity, где, как воспроизвести).
3. Автофикс критичных и высоких проблем.
4. Повторный ревью после фиксов.
5. Тестирование (unit/integration/smoke/e2e).
6. Обновление документации и статусов.
7. Повтор цикла, пока все критерии готовности не выполнены.

## 4. Ключевые требования к результату
- Ingestion стабильно индексирует документы из `drive-sync`.
- Новые/измененные файлы попадают в Qdrant.
- Удаленные файлы удаляются из Qdrant (replace semantics).
- Telegram бот отвечает на основе документов из целевой коллекции.
- Docker stack стартует после перезагрузки сервера без ручных костылей.
- Не остается блокирующих ошибок в логах сервисов.

## 5. Тестовая политика
- Минимум: unit + smoke + e2e по контуру ingestion->retrieval->answer.
- Целевой порог покрытия: 80% (фиксируется числом в отчете).
- Любой новый фикс должен иметь тест или обоснование, почему тест невозможен.

## 6. Реестр ошибок (обязательный формат)
Для каждой ошибки:
- `ID`
- `Severity` (`critical|high|medium|low`)
- `Компонент` (ingestion/bot/qdrant/docker/infra)
- `Симптом`
- `Корневая причина`
- `Фикс`
- `Статус` (`open|in_progress|fixed|verified`)

## 7. Трекинг статуса
Статусы задач:
- `todo`
- `in_progress`
- `blocked`
- `done`

Каждое изменение статуса должно быть зафиксировано в `CLAUDE.md`:
- что сделано;
- что осталось;
- какие новые риски/ошибки обнаружены.

## 8. Definition of Done
Задача считается завершенной только если одновременно:
- контур Google Drive -> Qdrant -> Telegram проверен end-to-end;
- критичные и high-ошибки закрыты и в статусе `verified`;
- тесты проходят, coverage зафиксирован;
- документация и реестр обновлены.

## 9. Формат финального отчета оркестратора
1. Что было сломано.
2. Что исправлено (по файлам и сервисам).
3. Что проверено (команды/тесты/результаты).
4. Что осталось (если есть) + конкретный риск.
5. Следующие шаги (при необходимости).
