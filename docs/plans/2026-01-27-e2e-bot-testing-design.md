# E2E Bot Testing Design

**Дата:** 2026-01-27
**Статус:** Draft
**Цель:** Полное E2E тестирование Telegram бота на живых данных с автоматической оценкой качества

---

## 1. Обзор

### Что тестируем

E2E (End-to-End) тесты проверяют весь путь от пользователя до ответа:

```
Telethon Userbot → Telegram API → Bot Server → Qdrant → LLM → Ответ
```

### Зачем нужно

| Unit tests говорят | E2E tests показывают |
|--------------------|----------------------|
| `VoyageService.embed_query()` работает | Embeddings реально ищут релевантные квартиры |
| `QdrantService.hybrid_search()` возвращает результаты | Фильтр "до 100к" реально фильтрует цены |
| `LLMService.stream_answer()` стримит текст | Ответ бота понятный и полезный для юзера |
| `classify_query()` определяет CHITCHAT | "Привет" реально отвечается без RAG |

### Ключевые решения

- **Библиотека:** Telethon (conversation API)
- **Оценка качества:** Claude как судья (LLM-as-Judge)
- **Тестовые данные:** Генерируем 100 фейковых квартир
- **Покрытие:** Все фичи бота (~25 сценариев)

## 3. Тестовые данные

### Проблема

Текущие данные в Qdrant — информационные статьи о Болгарии, а НЕ каталог недвижимости.
Поля `price`, `rooms`, `city`, `distance_to_sea` пустые.

### Решение

Генерируем 100 фейковых квартир с реалистичными данными.

### Структура данных

```python
@dataclass
class Property:
    id: str                    # UUID
    title: str                 # "2-комнатная квартира в Солнечный берег"
    description: str           # Детальное описание (200-500 слов)
    city: str                  # Город
    district: str              # Район/комплекс
    rooms: int                 # 0=студия, 1, 2, 3, 4+
    price: int                 # EUR
    area: int                  # м²
    floor: int                 # Этаж
    total_floors: int          # Этажность дома
    distance_to_sea: int       # Метры до моря
    year_built: int            # Год постройки
    features: list[str]        # ["бассейн", "паркинг", "вид на море"]
```

### Распределение (100 квартир)

| Параметр | Значения | Распределение |
|----------|----------|---------------|
| **Города** | Солнечный берег, Несебр, Бургас, Поморие, Святой Влас, Равда | 30/25/15/15/10/5 |
| **Комнаты** | Студия, 1, 2, 3, 4+ | 20/25/30/20/5 |
| **Цены** | 35,000 — 250,000 EUR | Нормальное распределение, медиана 85,000 |
| **Площадь** | 25 — 150 м² | Коррелирует с комнатами |
| **До моря** | 50 — 2000 м | Логнормальное (больше близких) |

### Пример описания

```
Уютная студия в комплексе "Сансет Резорт", Солнечный берег.
Полностью меблирована, кондиционер, балкон с видом на бассейн.
Комплекс с охраной 24/7, ресторан, спа-центр. До пляжа 150м.
Идеально для сдачи в аренду или отдыха. Годовые расходы ~400 EUR.
```

| # | Сценарий | Query | Ожидание |
|---|----------|-------|----------|
| 2.1 | Приветствие | `Привет!` | Быстрый ответ без RAG, дружелюбный тон |
| 2.2 | Благодарность | `Спасибо большое` | Вежливый ответ |
| 2.3 | Прощание | `До свидания` | Прощание |

| # | Сценарий | Query | Ожидание |
|---|----------|-------|----------|
| 3.1 | До X евро | `квартиры до 80000 евро` | Все цены ≤ 80000 |

| 3.4 | Без фильтра цены | `покажи квартиры` | Разные цены |

### Группа 4: Фильтры — Комнаты (4 теста)

| # | Сценарий | Query | Ожидание |
|---|----------|-------|----------|
| 4.1 | Студия | `студия` | Студии или 1-комнатные |
| 4.2 | 2-комнатные | `двухкомнатная квартира` | 2 комнаты |
| 4.3 | 3+ комнат | `трехкомнатные и больше` | 3+ комнат |
| 4.4 | Комбо | `2-комнатная до 100к` | 2 комнаты И цена ≤ 100к |

### Группа 5: Фильтры — Локация (3 теста)

| # | Сценарий | Query | Ожидание |
|---|----------|-------|----------|
| 5.1 | Город кириллица | `квартиры в Несебр` | Упоминание Несебр |
| 5.2 | Город транслит | `Sunny Beach` | Транслит → Солнечный берег |
| 5.3 | Расстояние до моря | `до 300м от моря` | Упоминание расстояния |

### Группа 6: Hybrid Search + Rerank (3 теста)

| # | Сценарий | Query | Ожидание |
|---|----------|-------|----------|
| 6.1 | Семантический | `уютная квартира с видом` | Релевантные результаты |
| 6.2 | Точный ID | `корпус 5 этаж 3` | Точное совпадение |
| 6.3 | Сложный | `2-комн в Солнечный берег до 120к с видом на море` | Все фильтры + семантика |

### Группа 7: Edge Cases (3 теста)

| # | Сценарий | Query | Ожидание |
|---|----------|-------|----------|
| 7.1 | Пустой результат | `замок за 1 евро` | "Ничего не нашел", предложение переформулировать |
| 7.2 | Очень длинный | `[100+ слов запрос]` | Корректный ответ или graceful error |
| 7.3 | Спецсимволы | `квартира <script>alert(1)</script>` | Безопасная обработка |

## 5. Claude Judge

### Критерии оценки (5 критериев, 0-10 баллов)

| Критерий | Вес | Описание |
|----------|-----|----------|
| **Relevance** | 30% | Ответ соответствует вопросу |
| **Completeness** | 25% | Достаточно информации, есть конкретика |
| **Filter Accuracy** | 20% | Фильтры применены корректно (цена, комнаты, город) |
| **Tone & Format** | 15% | Дружелюбный тон, Markdown форматирование |
| **No Hallucination** | 10% | Нет выдуманных данных, признаёт если не нашёл |

**Pass threshold:** total_score >= 6.0

### System Prompt

```markdown
Ты — судья качества ответов RAG-бота по недвижимости в Болгарии.

## Твоя задача
Оценить ответ бота по 5 критериям. Для каждого критерия дай балл 0-10.

## Критерии оценки

### 1. Relevance (релевантность) — 0-10
- 10: Ответ точно отвечает на вопрос
- 7-9: Ответ релевантен, но есть minor отклонения
- 4-6: Частично релевантен
- 0-3: Не отвечает на вопрос

### 2. Completeness (полнота) — 0-10
- 10: Вся необходимая информация, конкретные цены/адреса
- 7-9: Достаточно информации для принятия решения
- 4-6: Базовая информация, не хватает деталей
- 0-3: Слишком короткий или пустой ответ

### 3. Filter Accuracy (точность фильтров) — 0-10
- 10: Все упомянутые объекты соответствуют фильтрам запроса
- 7-9: Большинство соответствует, 1-2 отклонения
- 4-6: Половина соответствует
- 0-3: Фильтры проигнорированы
- N/A: Если в запросе нет фильтров

### 4. Tone & Format (тон и формат) — 0-10
- 10: Дружелюбный тон, хорошее Markdown форматирование
- 7-9: Адекватный тон, читаемый формат
- 4-6: Нейтральный, но сложно читать
- 0-3: Грубый тон или нечитаемый формат

### 5. No Hallucination (без галлюцинаций) — 0-10
- 10: Все факты можно проверить, признаёт незнание
- 7-9: Нет явных выдумок
- 4-6: Есть сомнительные утверждения
- 0-3: Явно выдуманные данные

## Формат ответа (JSON)
{
  "relevance": {"score": 8, "reason": "..."},
  "completeness": {"score": 7, "reason": "..."},
  "filter_accuracy": {"score": 9, "reason": "..."},
  "tone_format": {"score": 8, "reason": "..."},
  "no_hallucination": {"score": 10, "reason": "..."},
  "total_score": 8.2,
  "pass": true,
  "summary": "Краткий вердикт в 1-2 предложения"
}

**pass = true если total_score >= 6.0**
```

## 7. Запуск

```bash
# 1. Сгенерировать тестовые данные (один раз)

python scripts/index_test_properties.py

# 2. Запустить E2E тесты
python scripts/e2e/runner.py

# Опции:
python scripts/e2e/runner.py --group filters    # Только фильтры
python scripts/e2e/runner.py --scenario 3.1     # Один сценарий
python scripts/e2e/runner.py --parallel 3       # 3 параллельных теста
```

## 9. Требования

### Перед запуском

1. **Telegram API credentials** — получить на https://my.telegram.org
2. **Anthropic API key** — для Claude Judge
3. **Docker services** — Qdrant, Redis запущены
4. **Bot running** — `@test_nika_homes_bot` работает

### Best Practices (из Exa search)

- Telethon `conversation()` API для send/wait pattern
- Timeout 60s для streaming ответов
- Rate limiting между тестами (2s delay)
- Structured rubrics для LLM-as-Judge
