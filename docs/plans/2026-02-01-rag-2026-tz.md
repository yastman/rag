## 0) Контекст (что уже есть)

Система уже близка к "production-grade" по структуре:

- **Retrieval:** Qdrant `query_points()` + nested `Prefetch` + `Fusion(RRF)` (dense + sparse) в одном запросе.
- **Sparse:** BM42 (FastEmbed).
- **Rerank:** Voyage rerank (в боте) + кеширование результата rerank.
- **Cache:** RedisVL `SemanticCache` + `EmbeddingsCache` + дополнительные Tier-2 кеши (analysis/search/sparse/rerank/conversation).
- **Orchestration:** Query routing (CHITCHAT/SIMPLE/COMPLEX) в Telegram bot.
- **Infra:** Docker dev stack поднят, есть `dev-redis (8.4.0)` и `dev-qdrant (1.16)`.

Важно по Redis: **Redis 8.4** — это серверный движок (FT.*), а **RedisVL** — Python SDK/обвязка поверх redis-py,
который ускоряет реализацию кеш-слоёв (SemanticCache/EmbeddingsCache) и берёт на себя схему/индексы. В текущем
репозитории RedisVL уже используется и является “baseline” для кеш-слоя.

Болевые точки “под 2026”:

1) Qdrant коллекции создаются с **Scalar INT8 quantization**, но не с **Binary quantization**.
2) Redis поднят как 8.4, но политика eviction сейчас `allkeys-lfu` (опасно для индексов/служебных ключей).
3) В кеше используются несколько независимых Redis-клиентов (RedisVL + Tier-2), риск “connection churn” под нагрузкой.
4) Отсутствует “small-to-big” (late expansion/parent retrieval) для повышения качества ответа без увеличения top_k.
5) ACORN (фильтр-устойчивый поиск) пока не включён и может требовать обновления qdrant-client.

### 0.1 Матрица фич (Redis/Qdrant): "уже есть" vs "в плане"

Эта матрица нужна, чтобы не путать:

- “фича доступна в контейнере” ≠ “фича используется в коде”
- “фича используется в коде” ≠ “настроено по best practice 2026”

**Redis 8.4**

- ✅ Используется в коде (baseline):
  - RedisVL `SemanticCache` / `EmbeddingsCache` / `SemanticMessageHistory` (L1/L2 + conversation memory)
  - Tier-2 key/value кеши (analysis/search/sparse/rerank/conversation) через `redis.asyncio`
- ⚠️ Доступно в Redis 8.4, но не используется как “ключевая фича” (и это нормально):
  - `FT.HYBRID` (актуально только если переносим retrieval в Redis; сейчас retrieval в Qdrant)
- ❌ Нужно “донастроить” по best practice:
  - eviction policy: перейти с `allkeys-lfu` на `volatile-lfu`/`volatile-lru` + `maxmemory-samples=10`
  - единый connection pool на процесс + передача клиента/настроек в RedisVL компоненты

**Qdrant 1.16**

- ✅ Используется в коде (baseline):
  - `query_points()` + nested `Prefetch` + `Fusion(RRF)` для hybrid (dense+sparse) в одном запросе
  - query-time toggles для quantization (`ignore/rescore/oversampling`) как основа A/B
- ⚠️ Уже есть частично:
  - коллекции при индексации создаются с Scalar INT8 quantization (baseline), но это не binary
- ❌ В плане (best practice 2026):
  - Binary quantization (на уровне коллекции) + A/B против INT8
  - ACORN (conditional) — после проверки поддержки в `qdrant-client` и только для фильтрованных запросов
  - (optional) query-side/asymmetric encoding для binary как полировка, только как A/B параметр
  - HNSW профили (high-speed/balanced) с управляемым query-time `hnsw_ef`

### 0.3 Platform Standard 2026: uv-first (обязательно)

Для этого репозитория фиксируем **обязательный стандарт**: `uv` является единым инструментом для

## 3) Архитектура целевого решения

### 3.1 Target pipeline (высокоуровнево)

1) **Ingestion**
   - Docling → структурированный текст (Markdown/blocks)
   - Chunking (семантический) → *малые чанки* (поиск)
   - Overlap: **15–30%** (для edge-case на границах) либо компенсировать overlap’ом через small-to-big window
   - Сохранение связей: `doc_id`, `section_id`, `chunk_order`, `(start_offset, end_offset)`
2) **Indexing**
   - Dense embeddings (Voyage или локальный энкодер)
   - Sparse embeddings (BM42)
   - Payload: `text`, `doc_id`, `chunk_order`, `source`, `metadata`
3) **Retrieval**
   - Qdrant: dense + sparse → RRF fusion (native)
   - Опционально: ACORN для “жёстких фильтров”
4) **Rerank**
   - Rerank top-N (10–30 кандидатов), кеш и TTL
5) **Small-to-big expansion**
   - После rerank берём top-1..top-3 чанка и **расширяем окно** (например, ±2 соседних чанка в рамках doc_id)
6) **Generation**
   - LLM через LiteLLM gateway, с fallbacks
7) **Caching**
   - Слои L1–L5 (см. ниже), версия ключей

### 3.2 “Идеальный стек 2026” (теория → опции для экспериментов)

Этот проект тестовый, поэтому целевой результат — **референсная архитектура**, которая покрывает лучшие практики 2026
и позволяет безопасно экспериментировать через A/B.

**Базовый (Text-first) “идеальный” стек для этого репо:**

- **Parsing:** Docling (структура > “красивый текст”).
- **Chunking:** semantic chunking + **small-to-big / late expansion** (качество без раздувания retrieval).
- **Retrieval:** Qdrant hybrid (dense + sparse) + native fusion (RRF) + (опционально) ACORN при фильтрах.
- **Rerank:** отдельный слой rerank top-N (10–30) + кеширование.
- **Caching:** Redis 8.4 как multi-layer cache + строгие пороги/фильтры + eviction policy и pooling.
- **Routing:** query routing (chitchat/simple/complex) как cost-control.
- **Observability + Eval:** preflight + bench harness + минимум метрик (latency p95, cache hit-rate, recall proxy).

**Опциональные ветки 2026 (делать только как R&D, не смешивать с baseline):**

1) **Vision-first RAG:** страничные/визуальные эмбеддинги для PDF-страниц (полезно для схем/диаграмм).
2) **Agentic RAG:** multi-step retrieval, уточняющие подзапросы, проверка источников.
3) **Смена embedding/reranker моделей:** только через бенч/регресс-набор, без “миграции на веру”.
4) **Contextual chunking (Contextual Retrieval):** добавить к каждому чанку краткий “док-контекст” (один вызов LLM на документ
   или на секцию) как быстрый прирост качества retrieval; включать флагом и мерить A/B.

## 4) Redis 8.4: требования и best practices

### 4.1 Eviction policy и memory

**Проблема:** `allkeys-lfu` может выкидывать ключи без TTL (в т.ч. служебные/индексные), что приводит к деградации RedisVL SemanticCache и нестабильности.

**Требование (тестовый стандарт):**

- `maxmemory-policy`: **volatile-lfu** (или volatile-lru)
- `maxmemory-samples`: **10** (вместо дефолтного 5) для более стабильного LFU/LRU eviction на малых объёмах памяти
- Все “пользовательские” cache-записи (L1–L5) обязаны иметь TTL.
- Индексы/служебные ключи должны быть защищены косвенно: не иметь TTL и не подпадать под eviction policy.

**Практическое действие:** обновить `docker-compose.dev.yml` / `docker-compose.local.yml` (redis command) под `volatile-lfu`
и зафиксировать это как “правильно по умолчанию”.

### 4.2 Multi-layer caching (ключи, TTL, инвалидация)

Кэшируем только то, что даёт измеримый выигрыш, и **всегда** версионируем ключи.

**Версионирование:**

- `CACHE_SCHEMA_VERSION` (уже есть) — обязателен во всех ключах.
- Дополнительный `retrieval_version` (например, `qdrant_collection` + `dense_model` + `sparse_model`) — для L3/L4.

**Рекомендуемые слои:**

- **L1: Semantic (Query → Answer)**: 24–72h
  - Hit condition: cosine distance <= threshold (строго) + filters (user_id/lang/query_type)
- **L2: Embeddings (Text → Embedding)**: 7–30d
  - Hit condition: exact text hash
- **L3: Sparse embedding (Text → Sparse)**: 7–30d
- **L4: Retrieval results (Embedding+Filters → doc_ids)**: 1–24h
  - Важно: инвалидировать при смене коллекции/индекса
- **L5: Rerank results (QueryEmbedding + doc_ids → ranked doc_ids)**: 1–6h

### 4.3 FT.HYBRID (опционально)

FT.HYBRID включать только при одном из сценариев:

1) Небольшой корпус полностью живёт в Redis (и Redis становится retrieval engine).
2) Нужен hybrid поиск “прямо в кеше” как отдельный продуктовый режим.

В текущем проекте retrieval делает Qdrant, Redis — кеш. Поэтому FT.HYBRID не является приоритетом, но должен быть документирован как опция.
RedisVL при этом **может** использовать FT.HYBRID для hybrid retrieval (если мы решим переносить retrieval в Redis), но L1 semantic cache обычно
достаточно vector-threshold + фильтров.

### 4.4 Connection pooling / Redis clients

**Требование:** единый пул соединений на процесс.

- Tier-2 (`redis.asyncio`) должен использовать `ConnectionPool.from_url(...)`.
- RedisVL SemanticCache/EmbeddingsCache/SemanticMessageHistory должны получать **переданный redis_client/async_redis_client**, а не создавать свой.
- Добавить “preflight” метрики: число соединений, ошибок, evicted_keys.

**Baseline-решение для этого репо:** оставить RedisVL, но гарантировать, что:
- `CacheService` создаёт один `redis.asyncio.Redis` (через общий `ConnectionPool`)
- RedisVL компоненты получают этот клиент (или связанный sync client), чтобы не плодить соединения и не расходиться в настройках таймаутов.

## 5) Qdrant 1.16: требования и best practices

### 5.1 Коллекции и quantization (baseline vs binary)

Сделать две стратегии:

**A) Baseline (текущая):** Scalar INT8 quantization
**B) Candidate (2026):** Binary quantization + rescore + oversampling

**Требование:** поддержать A/B на одном и том же наборе чанков:

- либо две коллекции (`*_int8` и `*_binary`),
- либо одна коллекция, если конфиг позволяет переключаться без reindex (обычно нет).

**Search-time параметры (обязательны для binary):**
- `rescore=true`
- `oversampling=2.0` (и тестировать 2–4)

**Опционально (полировка, только при наличии поддержки в Qdrant + qdrant-client):**
- “asymmetric”/query-side encoding для бинарной квантизации (например, скалярное кодирование запроса), чтобы повысить recall
  при бинарном storage. Включать только как A/B-параметр и фиксировать в отчёте.

### 5.2 ACORN (условное включение)

ACORN включать только при “жёстких фильтрах” и низкой селективности.

**Требование:** добавить “селективность” как входной сигнал (эвристика):
- если фильтр содержит `user_id/tenant_id` и ожидаемо “узкий” → enable
- или если есть счётчик: `estimated_selectivity < 0.4` → enable

**Критическая зависимость (SDK):** ACORN включается через `SearchParams`, но для этого Python SDK должен поддерживать
соответствующие модели/поля. В этом репо ACORN считается “готовым к включению”, только если:

- В `qdrant_client.models` доступен тип/параметры ACORN (например, `AcornSearchParams`, либо эквивалентная структура),
- Либо SDK документированно поддерживает передачу ACORN в `search_params` без type wrappers.

**Проверка готовности (acceptance для обновления SDK):**

- Добавить preflight-проверку/скрипт: `python -c "from qdrant_client import models; print('AcornSearchParams' in dir(models))"`.
- Если `False` → обновить `qdrant-client` до версии, совместимой с сервером Qdrant `v1.16`, и повторить проверку.

ACORN должен быть под **feature-flag** и включаться только при наличии фильтров и/или низкой селективности.

### 5.3 HNSW tuning и on-disk

Сформировать 2 профиля:

- **Balanced:** `m=16`, `ef_construct=200`, `hnsw_ef=128`
- **High-speed:** `m=8`, `ef_construct=100`, `hnsw_ef=64`

`hnsw_ef` должен быть управляемым на запросе (через config), чтобы можно было A/B.

## 6) Small-to-big retrieval (late expansion)

### 6.1 Идея

Поиск делаем по маленьким чанкам (точность), а контекст в LLM отдаём “как из документа”:

- нашли top-1..top-3 чанка,
- расширили каждый: `chunk_order - W ... chunk_order + W` (W=1..3),
- склеили с дедупликацией и лимитом по токенам.

### 6.2 Требования к данным

Во всех чанках должны быть поля:

- `doc_id` (стабильный идентификатор документа/файла)
- `chunk_order` (порядок чанков внутри документа)
- `text`
- опционально: `page_range`, `section`, `offsets`

### 6.3 Acceptance criteria

- “Расширенный контекст” не дублирует куски, сохраняет порядок, и ограничен по размеру.
- На тестовом наборе запросов повышает “answer completeness” (ручная оценка) без увеличения top_k retrieval.

### 8.0 Feature Flags & Profiles (единый контракт)

Цель: все “топовые фичи 2026” должны включаться **опциями**, а не “навсегда в коде”, чтобы:
- быстро делать A/B;
- иметь аварийный rollback (выключить фичу без деплоя);
- не превращать систему в “комбайн”, где всё всегда включено.

**Рекомендация:** вместо простых `True/False` использовать режимы `off | on | auto` и профили.

**Профиль ретривала (единая ручка):**
- `retrieval_profile`: `fast | balanced | quality`
  - управляет: `top_k`, `prefetch_multiplier`, `rerank_top_n`, `hnsw_ef`, `quantization_*`

**Фичи как режимы:**
- `acorn_mode`: `off | on | auto`
  - `auto` включать **только** если есть фильтры и низкая селективность (см. Milestone D)
  - параметры: `acorn_enabled_selectivity_threshold`, `acorn_max_selectivity`
- `quantization_mode`: `off | scalar | binary`
  - параметры: `quantization_rescore`, `quantization_oversampling`
- `small_to_big_mode`: `off | on | auto`
  - `auto`: включать только для `query_type=COMPLEX` (или при низкой уверенности retrieval)
- `contextual_chunking_mode`: `off | on` (R&D, Milestone E)

**Минимальный набор ENV для управления (пример):**
- `RAG_RETRIEVAL_PROFILE=balanced`
- `RAG_ACORN_MODE=auto`
- `RAG_QUANTIZATION_MODE=scalar`
- `RAG_SMALL_TO_BIG_MODE=auto`

**Профили: рекомендуемые дефолты (можно переопределять ENV/конфигом)**

| Profile | Цель | `top_k` | `prefetch_multiplier` | `rerank_top_n` | `hnsw_ef` | `quantization_mode` | `quantization_rescore` | `quantization_oversampling` | `small_to_big_mode` | `small_to_big_window` |
| ------: | ---- | ------: | --------------------: | -------------: | --------: | ------------------: | ---------------------: | ---------------------------: | ------------------: | ---------------------: |
| `fast` | минимальная p95 | 8 | 2 | 10 | 64 | `scalar` | `true` | 2.0 | `off` | — |
| `balanced` | дефолт | 10 | 3 | 20 | 128 | `scalar` | `true` | 2.0 | `auto` | ±1 chunk |
| `quality` | максимум качества | 12 | 4 | 30 | 256 | `binary` | `true` | 3.0 | `on` | ±2 chunks |

Примечания:
- `quantization_mode=binary` предполагает отдельную Qdrant коллекцию с BinaryQuantization (Milestone B), иначе профиль `quality` должен деградировать в `scalar`.
- `small_to_big_mode=auto` рекомендуется включать только для `query_type=COMPLEX` (QueryRouter) и/или при низкой уверенности retrieval.

### 10.1 RAG Evaluation Metrics (КРИТИЧНО)

**Проблема:** В плане есть observability (Langfuse), но нет **систематической оценки качества RAG**.

**Best Practice 2025-2026:** Использовать RAG-специфичные метрики:

| Metric | Description | Tool |
| ------ | ----------- | ---- |
| **Faithfulness** | Ответ основан на retrieved context, а не на галлюцинациях | DeepEval, Ragas |
| **Answer Relevancy** | Ответ релевантен вопросу | DeepEval, Ragas |
| **Context Relevancy** | Retrieved chunks релевантны вопросу | LLM-as-a-Judge |
| **Context Recall** | Найдены все нужные документы | Ground truth dataset |
| **Context Precision** | Нет лишних документов в контексте | LLM-as-a-Judge |
| **Hallucination Score** | Процент неподтверждённых утверждений | MetaRAG pattern |

# Ragas + Langfuse (из документации)
from ragas.metrics import faithfulness, answer_relevancy, context_precision

### 10.2 Query Expansion / HyDE (РЕКОМЕНДОВАНО)

**Проблема:** Короткие/неточные запросы плохо матчатся с длинными документами.

**HyDE (Hypothetical Document Embeddings):**
1. LLM генерирует "гипотетический ответ" на вопрос
2. Embed этот ответ (а не оригинальный вопрос)
3. Поиск по embedding гипотетического ответа

**Результат:** +10-20% recall для неточных/коротких запросов.

```python
# Псевдокод HyDE
def hyde_embed(query: str) -> list[float]:
    hypothetical_doc = llm.generate(f"Write a document that answers: {query}")
    return embed(hypothetical_doc)  # embed the answer, not the question
```

**Когда использовать:**
- Короткие запросы (< 5 слов)
- Вопросы без ключевых слов из документов
- Zero-shot сценарии

**Задачи (Milestone G):**
- [ ] Добавить HyDE как опциональный query preprocessor
- [ ] A/B: baseline vs HyDE на коротких запросах
- [ ] Feature flag: `use_hyde=True` для определённых query_types

### 10.3 Guardrails (РЕКОМЕНДОВАНО)

**Проблема:** RAG может выдавать небезопасный/некорректный контент. В плане нет защиты.

**Input Guardrails:**
- Jailbreak detection (попытки обхода)
- PII detection (персональные данные в запросе)
- Topic filtering (off-topic запросы)

**Output Guardrails:**
- Hallucination detection (factoid decomposition + verification)
- Toxicity filtering
- Confidence scoring + fallback responses

**Паттерн MetaRAG (2025):**
1. Decompose ответ на atomic factoids
2. Для каждого factoid проверить: есть ли он в retrieved context?
3. Score = % подтверждённых factoids
4. Если score < threshold → fallback response

```python
# Пример guardrail
def check_grounding(answer: str, context: list[str]) -> float:
    factoids = extract_factoids(answer)  # LLM decomposition
    grounded = sum(1 for f in factoids if is_supported(f, context))
    return grounded / len(factoids)

if check_grounding(answer, context) < 0.7:
    return "Не могу дать точный ответ. Попробуйте уточнить вопрос."
```

**Задачи (Milestone H):**
- [ ] Добавить input validation (off-topic detection через QueryRouter)
- [ ] Добавить confidence scoring для ответов
- [ ] Реализовать fallback response при low confidence
- [ ] Опционально: NeMo Guardrails / Guardrails AI интеграция

### 10.4 Late Chunking (АЛЬТЕРНАТИВА contextual chunking)

**Проблема:** При обычном chunking теряется контекст на границах чанков.

**Late Chunking (Jina AI, 2024):**
1. Embed весь документ через long-context model (8K+ tokens)
2. Получить token-level embeddings
3. **После** embedding разбить на чанки (mean pooling по токенам)
4. Результат: каждый чанк "знает" про соседние чанки

**Преимущества:**
- Не требует LLM-вызовов (в отличие от contextual chunking)
- Сохраняет контекст между чанками
- Работает с любой long-context embedding моделью

**Ограничения:**
- Требует long-context embedding model (jina-embeddings-v3, Voyage-3-large)
- Voyage-4-large уже поддерживает 32K контекст

### 10.6 Query Decomposition для сложных запросов (ОПЦИОНАЛЬНО)

**Проблема:** Сложные multi-hop вопросы ("Сравни квартиры в корпусе 3 и 5 по цене") плохо работают с single-shot retrieval.

**Решение (2025 pattern):**
1. LLM разбивает сложный запрос на sub-queries
2. Каждый sub-query выполняется отдельно
3. Результаты объединяются для финального ответа

```python
# Псевдокод Query Decomposition
def decompose_query(query: str) -> list[str]:
    """LLM разбивает на под-запросы"""
    return llm.generate(f"Break this into simpler questions: {query}")

# Пример:
# "Сравни корпус 3 и 5" → ["Квартиры в корпусе 3", "Квартиры в корпусе 5"]
```

**Интеграция с QueryRouter:**
- `QueryType.COMPLEX` → включить decomposition
- `QueryType.SIMPLE` → single-shot

**Задачи (добавить в Milestone G или отдельный):**
- [ ] Добавить query decomposition для COMPLEX queries
- [ ] A/B: decomposition vs single-shot на multi-hop вопросах

### 10.7 Semantic Cache Tuning (УТОЧНЕНИЕ)

**Проблема:** Threshold для semantic cache (cosine distance) влияет на hit rate vs качество.

**Best Practice:**
